# JellyNews v1.0.3 — Notes de release

Date : 2026-07-06
Statut : validé après revue Makise

## Résumé utilisateur

JellyNews v1.0.3 corrige deux irritants d'exploitation : le navigateur recharge maintenant les nouveaux assets web après release grâce à un cache-buster simple basé sur la version applicative, et les campagnes newsletters sont envoyées plus proprement quand la liste d'abonnés grandit.

L'envoi SMTP reste volontairement KISS : pas de Redis, Celery ou service externe. JellyNews conserve un email distinct par abonné, ajoute un throttling par vagues configurable et refuse les campagnes concurrentes.

## Cache-buster applicatif

La version applicative est centralisée en `1.0.3`.

Les assets statiques modifiables chargés par les pages web portent une query string stable :

- `/static/style.css?v=1.0.3` ;
- `/static/app.js?v=1.0.3` ;
- SVG de marque référencés depuis les templates ou `style.css` quand leur nom reste constant.

Objectif : après déploiement v1.0.3, un navigateur qui a encore `style.css` ou `app.js` de v1.0.2 demande une URL différente et récupère la nouvelle version sans `Ctrl+Shift+R`.

Limites :

- le cache-buster couvre les assets servis par les templates web et les références statiques modifiées ;
- il ne purge pas un cache intermédiaire qui ignorerait les query strings ;
- les SVG référencés depuis `style.css` restent à surveiller lors du prochain bump si leur contenu change sans changement de nom.

## Envoi SMTP

La stratégie retenue est l'envoi individuel :

- un message par abonné ;
- un seul destinataire dans `To` ;
- aucun `Cc` ni `Bcc` ;
- header `List-Unsubscribe` et `List-Unsubscribe-Post` conservés quand `app_public_url` est configuré ;
- erreurs SMTP isolées par destinataire, les autres abonnés restent tentés.

Deux réglages admin contrôlent le throttling :

- `smtp_batch_size` : nombre de messages avant pause, défaut `25`, bornes `1..500` ;
- `smtp_batch_pause_seconds` : pause entre vagues, défaut `2`, bornes `0..3600`.

Ces réglages limitent les signaux anti-spam côté fournisseur SMTP. Ils ne remplacent pas une configuration DNS correcte : SPF, DKIM et DMARC doivent être configurés sur le domaine d'envoi.

### Stratégie anti-spam retenue

La CCI groupée est volontairement rejetée. Elle réduirait le nombre de connexions SMTP, mais elle empêche de personnaliser le lien de désinscription par abonné et produit un signal moins propre côté délivrabilité.

La stratégie v1.0.3 privilégie donc :

- une enveloppe simple, lisible par les fournisseurs SMTP ;
- un destinataire unique par message ;
- une version texte de repli ;
- un header de désinscription standard quand l'URL publique JellyNews est configurée ;
- une pause configurable entre vagues pour rester sous les seuils du fournisseur.

Limites :

- JellyNews ne contourne pas les quotas du fournisseur SMTP ;
- un domaine ou une IP avec mauvaise réputation restera pénalisé ;
- les gros volumes doivent être dimensionnés selon les limites horaires et journalières du relais utilisé ;
- le verrou de campagne est en mémoire et convient au lancement Uvicorn mono-worker actuel. Un déploiement multi-worker nécessiterait un verrou partagé.

## Prérequis de délivrabilité

Avant d'augmenter le nombre d'abonnés, vérifier :

- SPF autorisant le relais SMTP utilisé ;
- DKIM actif et aligné avec le domaine d'expédition ;
- DMARC publié, d'abord en observation puis durci progressivement ;
- réputation correcte de l'IP ou du relais SMTP ;
- adresse d'expéditeur cohérente et stable ;
- lien de désinscription fonctionnel via `app_public_url` ;
- volume d'envoi progressif si le domaine ou le relais est récent.

JellyNews améliore la forme de l'envoi ; la réputation reste celle de l'infrastructure mail.

## Nouveaux réglages admin

Deux champs SMTP sont exposés dans l'administration :

| Réglage | Défaut | Bornes | Effet |
|---|---:|---:|---|
| `smtp_batch_size` | `25` | `1..500` | Nombre de messages envoyés avant une pause. |
| `smtp_batch_pause_seconds` | `2` | `0..3600` | Durée de pause entre deux vagues, en secondes. |

Une valeur hors bornes est rejetée à la sauvegarde des paramètres. La valeur `0` pour la pause désactive l'attente entre vagues sans revenir à une CCI groupée.

## Envoi manuel

`POST /api/send-now` répond immédiatement avec `{"status":"queued","queued":true}` quand la campagne est lancée. Le travail continue en arrière-plan dans le processus FastAPI.

Si une campagne est déjà en cours, l'endpoint répond en HTTP 409 et ne lance pas de second envoi.

## Logs et archives

Les logs conservent :

- le nombre total de médias ;
- le nombre d'emails envoyés avec succès ;
- un détail synthétique des erreurs SMTP partielles, avec adresses masquées ;
- le statut `ok`, `partial`, `error` ou `skipped`.

Les archives continuent de stocker la version navigateur de la newsletter, sans lien de désinscription.

## Compatibilité

- Aucun changement de schéma SQLite.
- Les exports complets restent au schéma `2`.
- Le nom de fichier d'export courant devient `jellynews-backup-v1.0.3-secrets.json`.
- Les anciens exports v1.0.1/v1.0.2 restent importables.
- Aucun nouveau service externe n'est requis.

## Validation réalisée

- Suite unitaire complète.

## Rollback

Avant déploiement, conserver une copie du volume `data/`.

En cas de rollback applicatif :

1. Arrêter v1.0.3.
2. Restaurer l'image précédente.
3. Restaurer le volume `data/` si nécessaire.
4. Vérifier que l'interface recharge bien les assets de la version restaurée ; si un proxy frontal force son propre cache, purger ce cache côté proxy.
5. Ne pas importer un export v1.0.3 dans une version plus ancienne sans sauvegarde préalable du volume `data/`.
