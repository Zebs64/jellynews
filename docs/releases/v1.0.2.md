# JellyNews v1.0.2 — Notes de release

Date : 2026-07-06
Statut : validé après revue Makise

## Résumé utilisateur

JellyNews v1.0.2 transforme la sauvegarde applicative en véritable filet de sécurité : l'export JSON ne se limite plus à la configuration, il emporte aussi les abonnés, l'historique des envois et les archives de newsletters. L'import correspondant restaure ces catégories par fusion, sans supprimer les données locales existantes.

La version apporte aussi la refonte visuelle « Media Current » : interface sombre plus proche de Jellyfin, navigation plus lisible, logo SVG, états vides illustrés, tables responsives et panneau dédié à la sauvegarde complète.

## Sauvegarde complète

Endpoint : `GET /api/settings/export`

Le fichier exporté contient :

- `schema_version: 2` ;
- `app_version` ;
- `exported_at` en ISO UTC ;
- `settings` : configuration complète filtrée par les clés connues ;
- `subscribers` : adresses abonnées et dates de création ;
- `send_logs` : historique des envois ;
- `archives` : archives HTML des newsletters envoyées.

Le nom de fichier généré est `jellynews-backup-v1.0.2-secrets.json` pour rappeler que l'export contient des secrets applicatifs.

### Données volontairement exclues

L'export n'inclut pas :

- les comptes administrateurs ;
- les hashes de mots de passe ;
- `secret.key` ;
- la base SQLite brute ;
- les fichiers uploadés, dont le logo personnalisé.

Le fichier de sauvegarde doit être stocké comme un secret : mêmes protections qu'un dump contenant des clés SMTP, Jellyfin et LLM.

## Import et compatibilité

Endpoint : `POST /api/settings/import`

L'import v1.0.2 restaure par fusion :

- les paramètres connus ;
- les abonnés, avec emails normalisés en minuscules et doublons ignorés ;
- les logs d'envoi ;
- les archives de newsletters.

La compatibilité avec les anciens exports v1.0.1 est conservée : un JSON de configuration sans `schema_version` est traité comme un export settings-only. Dans ce cas, seuls les paramètres reconnus sont importés ; abonnés, logs et archives restent inchangés.

### Limites et validations

- Les clés inconnues de configuration sont ignorées.
- Les fuseaux horaires invalides sont rejetés en HTTP 400.
- Les emails invalides sont rejetés en HTTP 400.
- Les entrées d'historique ou d'archive mal formées sont rejetées en HTTP 400.
- Les imports ne suppriment pas les données locales absentes du fichier.
- Les logs et archives sont dédupliqués sur les colonnes métier exportées, faute de contrainte `UNIQUE` historique en base.
- Aucun plafond explicite de taille ou de nombre d'entrées importées n'est ajouté dans cette version ; ce point reste un durcissement possible ultérieur.

## Points techniques

- Version client Jellyfin : `1.0.2`.
- Sauvegarde schema v2 via `database.export_backup()`.
- Import préparé par validation dans `app/routers/api.py`, puis fusionné par `database.import_backup_data()`.
- Archives HTML importées échappées avant consultation pour éviter de resservir un HTML tiers comme contenu exécutable de même origine.
- Tests automatisés ajoutés pour export complet, import legacy, idempotence, rejet des historiques invalides et contrôle XSS sur archive importée.
- La compatibilité CSV abonnés est conservée pour les opérations rapides sur la liste d'abonnés.

## Rollback conseillé

Avant déploiement :

1. Arrêter le conteneur JellyNews.
2. Copier le volume persistant `data/` hors du chemin de production.
3. Conserver séparément l'image Docker ou le tag applicatif v1.0.1 si le retour arrière doit être possible.
4. Démarrer v1.0.2 et générer un export JSON de contrôle après validation.

En cas de rollback :

1. Arrêter v1.0.2.
2. Restaurer le volume `data/` sauvegardé avant migration.
3. Redémarrer l'image précédente.
4. Ne pas importer dans v1.0.1 un export v1.0.2 : l'ancien import ne connaît pas les sections `subscribers`, `send_logs` et `archives`.
