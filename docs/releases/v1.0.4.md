# JellyNews v1.0.4 — Notes de release

Date : 2026-07-07
Statut : validé après revue Makise

## Résumé utilisateur

JellyNews v1.0.4 rend l'administration plus lisible dès l'ouverture : après connexion, l'application affiche maintenant une page Accueil plutôt que de déposer l'utilisateur directement dans la configuration Jellyfin.

Cet accueil rassemble les indicateurs utiles avant une campagne : prochain envoi planifié, nombre de nouveautés détectées sur la période configurée, abonnés actifs et dernier envoi connu. La navigation reste volontairement simple : le logo JellyNews et le bouton Accueil ramènent au même panneau.

La release corrige également un défaut visuel de la sidebar : le bouton actif conserve son état lisible et son focus clavier, mais le trait vertical parasite a été supprimé.

## Accueil dashboard par défaut

Le panneau Accueil devient le premier écran affiché par `/` après authentification. Il sert de tableau de bord d'exploitation léger, sans introduire de service externe ni de nouvelle persistance.

Les quatre cartes exposées sont :

| Carte | Source | Rôle |
|---|---|---|
| Prochain envoi | planificateur APScheduler | confirmer que la planification hebdomadaire est active ou désactivée. |
| Nouveautés trouvées | client Jellyfin, période `lookback_days` | estimer le contenu potentiel de la prochaine newsletter. |
| Abonnés actifs | table abonnés SQLite | vérifier rapidement le volume de diffusion courant. |
| Dernier envoi | logs d'envoi SQLite | connaître le dernier statut, le nombre de médias et les destinataires traités. |

Si Jellyfin n'est pas configuré ou répond en erreur, l'accueil garde un état explicite : la carte des nouveautés indique l'indisponibilité au lieu d'exposer une exception technique à l'administrateur.

## Navigation

Deux accès mènent désormais au panneau Accueil :

- le premier bouton de la sidebar, libellé `Accueil` ;
- le logo JellyNews, rendu cliquable avec un libellé accessible `Retour à l'accueil JellyNews`.

Le même mécanisme JavaScript d'activation de panneau est utilisé pour les boutons de navigation et pour le logo. L'objectif est de conserver une interface KISS : pas de route supplémentaire, pas de rechargement complet, seulement un changement de panneau côté client.

## API d'administration

Un endpoint d'administration a été ajouté :

```http
GET /api/dashboard-summary
```

Il renvoie un résumé agrégé pour l'accueil : prochain run, période d'analyse, nombre de nouveautés, éventuelle erreur Jellyfin, nombre d'abonnés et dernier log d'envoi.

La route est montée sur le router `/api`, qui applique déjà la dépendance d'authentification globale. Elle n'est donc pas publique : un appel non authentifié répond en HTTP 401.

## Sécurité et données personnelles

Aucune métrique n'est exposée sans session administrateur.

Les données affichées restent agrégées : nombre d'abonnés, statut du dernier envoi, compteurs de médias et de destinataires. L'accueil n'affiche pas la liste des emails abonnés, les clés Jellyfin, les secrets SMTP, les clés LLM, le contenu des exports JSON ni le détail brut des erreurs.

Futaba a validé la release en amont d'après le handoff de revue transmis à Makise. Les limites habituelles restent inchangées : les exports complets contiennent des secrets et doivent être traités comme tels ; le volume `data/` reste la pièce sensible à protéger ; les routes `/api` supposent une session administrateur valide.

## Correction CSS sidebar active

La correction retire le pseudo-élément `.nav-btn.active::before`, responsable du trait vertical parasite dans la sidebar.

L'état actif n'est pas supprimé : il reste représenté par le gradient, la couleur de texte et la bordure du bouton actif. Le focus clavier reste visible via `:focus-visible` et la variable de focus existante, afin de ne pas dégrader l'accessibilité de la navigation.

## Cache-buster et version

La version applicative passe à `1.0.4`.

Les assets web modifiables continuent d'être référencés avec la query string issue de la version applicative :

- `/static/style.css?v=1.0.4` ;
- `/static/app.js?v=1.0.4` ;
- assets de marque référencés par les templates ou la feuille CSS lorsque le nom de fichier reste stable.

Le nom d'export complet suit également la version courante : `jellynews-backup-v1.0.4-secrets.json`.

## Compatibilité

- Aucun changement de schéma SQLite.
- Aucun nouveau service externe.
- Aucun changement de configuration Docker requis.
- Le modèle FastAPI monolithique reste inchangé.
- Les exports complets restent au schéma déjà documenté.
- Les anciens exports compatibles restent importables selon les règles existantes.

## Vérifications opérateur conseillées

Après déploiement :

1. se connecter à l'administration ;
2. vérifier que `/` affiche le panneau Accueil par défaut ;
3. cliquer sur un autre panneau, puis sur le logo JellyNews, et confirmer le retour à Accueil ;
4. vérifier que le bouton actif de sidebar ne montre plus de trait vertical parasite ;
5. confirmer que le focus clavier reste visible en naviguant au clavier ;
6. vérifier que `/api/dashboard-summary` répond en HTTP 401 hors session et en HTTP 200 sous session administrateur.

## Rollback

Avant déploiement, conserver une copie du volume `data/`.

En cas de retour arrière :

1. arrêter JellyNews v1.0.4 ;
2. restaurer l'image ou le tag applicatif précédent ;
3. restaurer le volume `data/` si le rollback accompagne une restauration d'état ;
4. purger le cache navigateur ou proxy si `style.css`, `app.js` ou les assets de marque v1.0.4 ont déjà été servis ;
5. vérifier que l'interface restaurée recharge bien les assets de la version précédente.
