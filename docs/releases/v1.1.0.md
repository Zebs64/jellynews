# JellyNews v1.1.0 — Notes de release

Date : 2026-07-07

## Résumé

JellyNews v1.1.0 ajoute une personnalisation contrôlée du rendu newsletter sans transformer l'admin en éditeur HTML libre. Le template historique reste le défaut `Classique`, et les installations existantes conservent donc un rendu compatible sans action manuelle.

Les trois nouveaux rendus applicatifs reprennent la direction Aï décrite dans `docs/design/jellynews-v1.1.0-email-templates-direction.md` : variation de hiérarchie, densité et rythme visuel, sans asset externe obligatoire ni CSS fragile pour les clients mail.

## Nouveautés

- Template `Classique` conservé comme rendu historique et valeur par défaut.
- Nouveaux templates email-safe :
  - `Courant éditorial` : rendu magazine avec premier média plus visible ;
  - `Catalogue compact` : rendu dense pour semaines chargées ;
  - `Affiche de séance` : rendu plus scénique pour programmation courte ou moyenne.
- Panneau `Apparence / Newsletter` :
  - sélection du template ;
  - cartes miniatures HTML/CSS côté admin ;
  - blocs contrôlés ;
  - activation/désactivation du bloc optionnel `intro` ;
  - réordonnancement ↑ ↓ des blocs déplaçables ;
  - reset vers les blocs par défaut ;
  - accès direct à la preview.
- Preview admin rendue via le même moteur serveur que l'envoi.
- Email de test SMTP rendu avec la newsletter réelle, envoyé à un seul destinataire validé.
- Sauvegarde JSON complète enrichie avec :
  - `newsletter_template_id` ;
  - `newsletter_blocks_json`.

## Fonctionnement applicatif

La configuration v1.1.0 repose sur un registre serveur fixe dans `app/services/newsletter_templates.py` :

| Template | Identifiant | Usage recommandé |
|---|---|---|
| `Classique` | `classic` | rendu historique, défaut rétrocompatible. |
| `Courant éditorial` | `editorial` | mise en avant plus magazine, premier média de section plus visible. |
| `Catalogue compact` | `compact` | semaines chargées, densité et lecture rapide. |
| `Affiche de séance` | `showcase` | programmation courte ou moyenne avec présentation plus scénique. |

L'interface admin consomme `GET /api/newsletter/templates` pour afficher les templates, les cartes miniatures et la liste des blocs. L'enregistrement passe par les paramètres `newsletter_template_id` et `newsletter_blocks_json`, puis par la validation serveur avant persistance SQLite.

Les blocs connus sont `preheader`, `header`, `intro`, `media_sections` et `footer`. `intro` est optionnel ; les autres blocs sont obligatoires. Le serveur normalise l'ordre pour conserver `preheader` en premier et `footer` en dernier, même si l'interface permet de déplacer les blocs intermédiaires.

## Compatibilité et sécurité

- Les anciennes sauvegardes sans configuration newsletter restent compatibles : JellyNews retombe sur `classic` + blocs par défaut.
- Les imports avec template inconnu, bloc inconnu, doublon ou bloc obligatoire manquant/désactivé sont rejetés en HTTP 400 avant persistance.
- La sélection de template ne vient jamais d'un chemin utilisateur : elle passe par un registre Python fixe.
- Aucun HTML libre n'est stocké ou rendu depuis l'admin.
- Les templates email restent sur tables HTML, CSS inline, sans JavaScript, sans CSS externe critique, sans webfont.
- Le lien `List-Unsubscribe` et le lien footer restent gérés par le mailer quand `app_public_url` est configurée.

## Preview et email de test

La preview admin et l'email de test utilisent désormais le même moteur Jinja2 que l'envoi réel. Si Jellyfin n'est pas configuré, indisponible ou ne renvoie aucune nouveauté, JellyNews affiche un échantillon contrôlé au lieu de bloquer la validation visuelle.

L'email de test valide l'adresse cible, rend la newsletter complète et l'envoie comme message unique. Il ne modifie pas les abonnés, ne lance pas de campagne et ne contourne pas les garde-fous SMTP existants.

## Export / import

Le fichier complet est nommé `jellynews-backup-v1.1.0-secrets.json` et contient toujours des secrets applicatifs nécessaires au fonctionnement (SMTP, Jellyfin, LLM). Il ne contient toujours pas les comptes administrateurs, `secret.key`, la base SQLite brute ni les fichiers uploadés.

## Validation conseillée

1. Ouvrir `Apparence / Newsletter`.
2. Sélectionner chaque template puis lancer une preview.
3. Désactiver/réactiver `Introduction IA`, réordonner les blocs autorisés, puis reset.
4. Envoyer une newsletter test vers une adresse de validation.
5. Exporter la sauvegarde complète, réimporter sur instance de test, vérifier que le template et les blocs sont conservés.

## Validation staging v1.1.0

La documentation a été finalisée après validation staging par l'utilisateur, revue Makise approuvée et verdict sécurité Futaba approuvé. La sandbox staging a ensuite été retirée volontairement ; aucun tag ni push de publication n'est réalisé par cette étape documentaire.

## Rollback

Avant mise à jour, sauvegarder le volume `data/`. En cas de rollback vers v1.0.4 :

1. arrêter JellyNews v1.1.0 ;
2. restaurer le volume `data/` sauvegardé avant migration ;
3. relancer l'image v1.0.4 ;
4. purger le cache navigateur/proxy si `style.css`, `app.js` ou les assets de marque v1.1.0 ont déjà été servis.

Ne pas importer une sauvegarde v1.1.0 dans une v1.0.4 : l'ancien serveur ignore les nouveaux réglages et ne sait pas les valider.
