# JellyNews v1.1.1 — Notes de release

Date : 2026-07-07

## Résumé

JellyNews v1.1.1 ajoute un diagnostic SMTP détaillé pour aider l’administrateur à comprendre pourquoi un message n’a pas été accepté par le serveur SMTP, sans prétendre fournir une preuve de livraison en boîte de réception.

## Nouveautés

- Diagnostic SMTP structuré côté backend : classe d’erreur, code SMTP, message SMTP neutralisé, catégorie, aide administrateur et caractère réessayable si disponible.
- Mappings opérationnels ajoutés pour les cas courants : `550 5.7.1`, `554 5.7.1`, `552 5.3.4`, `451 4.7.0`, `421`, plus règles générales `4xx` temporaires et `5xx` permanentes.
- Historique admin enrichi avec un sous-bloc lisible dans la colonne détail, échappé côté frontend et tronqué côté backend/frontend.
- Persistance SQLite enrichie via colonnes idempotentes dans `send_logs` : `error_class`, `smtp_code`, `smtp_error`, `smtp_category`, `smtp_hint`, `retryable`.
- Export/import de sauvegarde compatible : les anciens exports sans champs SMTP restent importables, les nouveaux conservent les diagnostics.
- Formulations corrigées : JellyNews parle de messages “acceptés par le serveur SMTP”, pas de messages “livrés”.

## Lecture des diagnostics SMTP

Chaque échec SMTP journalisé peut maintenant porter les champs suivants :

| Champ | Signification |
|---|---|
| `error_class` | classe Python de l’erreur SMTP ou réseau capturée. |
| `smtp_code` | code numérique renvoyé par le serveur SMTP, quand il existe. |
| `smtp_error` | message SMTP distant neutralisé et tronqué. |
| `smtp_category` | catégorie opérationnelle calculée par JellyNews. |
| `smtp_hint` | indication administrateur pour la correction ou le ralentissement. |
| `retryable` | `true` si l’échec paraît temporaire, `false` s’il paraît permanent, `null` si JellyNews ne peut pas conclure. |

Les principaux cas connus sont interprétés ainsi :

| Code | Catégorie | Lecture opérationnelle |
|---|---|---|
| `550 5.7.1` | `rejet_policy_spam` | rejet anti-spam, politique serveur ou réputation expéditeur ; vérifier SPF/DKIM/DMARC, réputation IP/domaine et contenu. |
| `554 5.7.1` | `contenu_policy_refuse` | contenu refusé, réputation ou règle de politique ; vérifier contenu, liens et authentification domaine. |
| `552 5.3.4` | `message_trop_gros` | message trop volumineux ; réduire images intégrées, nombre de médias ou poids HTML. |
| `451 4.7.0` | `rate_limit_greylisting` | limite temporaire, greylisting ou ralentissement requis ; augmenter la pause entre vagues SMTP. |
| `421` | `service_temporaire_throttle` | service indisponible, quota ou throttling temporaire ; réessayer plus tard et réduire le débit. |

Règle générale : les codes `4xx` sont temporaires et potentiellement réessayables après délai ; les codes `5xx` sont permanents ou liés à une politique serveur et exigent une correction avant nouvel essai.

## Acceptation SMTP vs livraison inbox

La métrique d’envoi reste volontairement formulée en “message(s) accepté(s) par le serveur SMTP”. Elle signifie que le relais SMTP configuré a pris le message en charge. Elle ne prouve pas que le message est arrivé dans la boîte de réception finale, qu’il n’a pas été classé en spam ou qu’un relais aval ne l’a pas rejeté ensuite.

JellyNews ne traite pas les bounces entrants, ne lit pas de DSN et ne dispose pas d’un retour inbox. La délivrabilité réelle dépend toujours du domaine d’envoi, de SPF/DKIM/DMARC, de la réputation IP/domaine, des quotas du relais, du contenu et des politiques du fournisseur destinataire.

## Sécurité et confidentialité

- Les messages SMTP distants sont traités comme hostiles : CR/LF et caractères de contrôle normalisés, troncature à 500 caractères pour `smtp_error` et `smtp_hint`.
- Les adresses email détectées dans les erreurs SMTP sont masquées, par exemple `v***@example.test`.
- Les motifs de secrets évidents (`password`, `token`, `secret`, `api_key`) sont redacted avant stockage.
- L’interface admin échappe les champs SMTP avant injection HTML et ne rend pas de HTML libre.

## Compatibilité

- La migration SQLite ajoute les nouvelles colonnes au démarrage si elles manquent.
- Les réglages `smtp_batch_size` et `smtp_batch_pause_seconds` restent configurables et validés avec les bornes existantes.
- L’envoi SMTP reste individuel : un `To` unique par message, aucun `Cc`/`Bcc`, et `List-Unsubscribe` conservé quand `app_public_url` est configurée.

## Limites connues

- Pas de réception des bounces entrants ni de DSN.
- Pas de preuve d’arrivée en inbox.
- Pas de stratégie de retry automatique avancée ; `retryable` qualifie l’erreur pour l’administrateur, mais ne planifie pas seul un nouvel envoi.
- Le diagnostic persistant retient le premier échec SMTP détaillé de la campagne afin de garder un log synthétique ; le détail textuel peut contenir jusqu’aux premiers échecs affichés.

## Validation conseillée

1. Provoquer une erreur SMTP de test avec un code connu, par exemple `550 5.7.1`, puis vérifier l’historique admin.
2. Vérifier que le détail affiche classe, code, catégorie, indication temporaire/permanente et aide.
3. Exporter puis réimporter une sauvegarde contenant un log SMTP enrichi.
4. Vérifier que les anciens exports sans champs SMTP s’importent toujours.

## Rollback

Avant mise à jour, sauvegarder le volume `data/`. En cas de rollback vers v1.1.0 :

1. arrêter JellyNews v1.1.1 ;
2. restaurer le volume `data/` sauvegardé avant migration si l’ancienne image ne tolère pas les colonnes supplémentaires ;
3. relancer l’image v1.1.0 ;
4. purger le cache navigateur/proxy si `style.css`, `app.js` ou les assets versionnés v1.1.1 ont déjà été servis.
